# Feature: open_account

## Summary
Open a new retail bank account for one individual customer and return a generated account identifier.

## Purpose
This feature creates the account that later money movements and reads depend on.

Without an opened account, the system cannot accept deposits or return a balance.

## System boundary

### Owns
- account opening
- generated `account_id`
- acceptance of the applicant data required for this example
- initial account status
- initial currency for this example
- returned success payload
- validation of the input contract

### Does not own
- customer master data beyond this request
- identity verification workflow
- AML or sanctions checks
- card issuance
- overdraft rules
- account closure
- deposits
- withdrawals
- transfers
- balance reads

## Real-world anchor
This example models a small retail account-opening flow for one individual customer. It keeps the request deliberately lean and leaves compliance workflows, product configuration, and verification steps outside the feature.

## Trigger
A client submits an account-opening request.

## Interface shape
- Method and path: `POST /accounts`
- Success statuses: `201 Created`
- Failure statuses: `400 Bad Request`, `422 Unprocessable Entity`, `500 Internal Server Error`

### Request body

```json
{
  "first_name": "Anna",
  "last_name": "Schneider",
  "date_of_birth": "1992-04-18",
  "residential_address": {
    "street_line_1": "Kastanienallee 15",
    "street_line_2": "",
    "postal_code": "10435",
    "city": "Berlin",
    "country_code": "DE"
  },
  "government_id": "D1234567X"
}
````

### Success response

Status: `201 Created`

```json
{
  "account": {
    "account_id": "acc_01J0ABCDEF1234567890XYZ",
    "status": "open",
    "currency": "EUR",
    "created_at": "2026-04-08T12:00:00Z",
    "holder": {
      "first_name": "Anna",
      "last_name": "Schneider",
      "date_of_birth": "1992-04-18"
    }
  }
}
```

### Error response shape

```json
{
  "error": {
    "code": "FIRST_NAME_REQUIRED",
    "message": "first_name is required."
  }
}
```

## Input contract

### Required fields

* `first_name`: string
* `last_name`: string
* `date_of_birth`: date
* `residential_address`: object
* `residential_address.street_line_1`: string
* `residential_address.postal_code`: string
* `residential_address.city`: string
* `residential_address.country_code`: string
* `government_id`: string

### Optional fields

* `residential_address.street_line_2`: string

### Field rules

#### `first_name`

* required
* non-empty after trimming
* maximum length: 100

#### `last_name`

* required
* non-empty after trimming
* maximum length: 100

#### `date_of_birth`

* required
* ISO date format: `YYYY-MM-DD`
* must be parseable as a calendar date
* must be a past date
* applicant must be at least 18 years old

#### `residential_address.street_line_1`

* required
* non-empty after trimming
* maximum length: 200

#### `residential_address.street_line_2`

* optional
* may be empty
* maximum length: 200

#### `residential_address.postal_code`

* required
* non-empty after trimming
* maximum length: 32

#### `residential_address.city`

* required
* non-empty after trimming
* maximum length: 120

#### `residential_address.country_code`

* required
* exactly 2 alphabetic characters after trimming
* stored uppercase

#### `government_id`

* required
* non-empty after trimming
* maximum length: 64

## Output contract

The response returns the authoritative created account payload.

### Response fields

#### `account.account_id`

* opaque string identifier
* generated by the system
* stable for the life of the account

#### `account.status`

* always `open` on creation

#### `account.currency`

* always `EUR` in this example

#### `account.created_at`

* UTC timestamp in RFC 3339 format

#### `account.holder`

* echoes the stored holder data used in this example

## Business rules

1. The system generates the `account_id`.
2. The account opens in status `open`.
3. The account opens with currency `EUR`.
4. The initial balance is zero.
5. This feature opens exactly one account per request.
6. This feature does not perform compliance review or external identity verification.
7. This feature does not decide product variants, pricing, or overdraft settings.
8. The same person may hold more than one account in this example.

## Preconditions

* valid input

## State effect

On success, the system creates one new account with:

* a generated `account_id`
* status `open`
* currency `EUR`
* holder data from the request
* initial balance zero

## Postconditions

* the account exists
* later `deposit_money` may add money to it
* later `fetch_balance` returns zero until a deposit succeeds

## Idempotency and duplication policy

This feature is not idempotent.

The request is not deduplicated by applicant name, date of birth, address, or government_id. In this example, repeated valid requests may open multiple accounts.

Safe retry handling is outside this feature.

## Failure behavior

### `400 Bad Request`

Use when:

* the request body is malformed JSON
* a field has the wrong type
* `date_of_birth` cannot be parsed in the required format

Example codes:

* `INVALID_JSON`
* `INVALID_FIELD_TYPE`
* `INVALID_DATE_FORMAT`

### `422 Unprocessable Entity`

Use when:

* `first_name` is blank
* `last_name` is blank
* `date_of_birth` is not in the past
* applicant is younger than 18
* `residential_address.street_line_1` is blank
* `residential_address.postal_code` is blank
* `residential_address.city` is blank
* `residential_address.country_code` is invalid
* `government_id` is blank

Example codes:

* `FIRST_NAME_REQUIRED`
* `LAST_NAME_REQUIRED`
* `DATE_OF_BIRTH_INVALID`
* `APPLICANT_MUST_BE_ADULT`
* `ADDRESS_REQUIRED`
* `COUNTRY_CODE_INVALID`
* `GOVERNMENT_ID_REQUIRED`

### `500 Internal Server Error`

Use when:

* the account cannot be persisted
* account creation fails due to an internal runtime problem outside the request contract

Example codes:

* `INTERNAL_ERROR`

### Example validation failure

```json
{
  "error": {
    "code": "APPLICANT_MUST_BE_ADULT",
    "message": "Applicant must be at least 18 years old."
  }
}
```

### Example internal failure

```json
{
  "error": {
    "code": "INTERNAL_ERROR",
    "message": "The account could not be opened due to an internal error."
  }
}
```

## Policy decisions

* `account_id` is generated by the slice
* account currency is fixed to `EUR`
* the feature opens one simple retail account type only
* repeated applicant data is allowed to produce another account
* no fuzzy duplicate detection
* no compliance workflow in this example
* no customer entity is modeled separately in this feature

## Data ownership

This feature owns:

* `account_id`
* account status
* account currency
* `created_at`
* holder data stored for this example

Other features may reference the account, but they do not own these fields.

## Observability

At minimum, log:

* request accepted
* account created
* validation failure
* internal failure
* generated `account_id`
* result
* error_code

Suggested log fields:

* `feature=open_account`
* `account_id`
* `result`
* `error_code`

Do not log full `government_id` by default.

## Test cases

1. Happy path: valid applicant data returns `201` and a generated `account_id`.
2. Validation failure: blank `first_name` returns `422 FIRST_NAME_REQUIRED`.
3. Validation failure: blank `last_name` returns `422 LAST_NAME_REQUIRED`.
4. Validation failure: malformed `date_of_birth` returns `400 INVALID_DATE_FORMAT`.
5. Validation failure: future or non-past `date_of_birth` returns `422 DATE_OF_BIRTH_INVALID`.
6. Validation failure: applicant younger than 18 returns `422 APPLICANT_MUST_BE_ADULT`.
7. Validation failure: blank address field returns `422 ADDRESS_REQUIRED`.
8. Validation failure: invalid `country_code` returns `422 COUNTRY_CODE_INVALID`.
9. Validation failure: blank `government_id` returns `422 GOVERNMENT_ID_REQUIRED`.
10. Bad request: malformed JSON returns `400 INVALID_JSON`.
11. Bad request: wrong field type returns `400 INVALID_FIELD_TYPE`.
12. Edge case: a second identical request creates a second account because this feature is not idempotent.
13. Edge case: `country_code` is normalized to uppercase.
14. Success response: assert the full response payload shape, including holder fields and RFC 3339 `created_at`.
15. Internal failure: simulated persistence failure returns `500 INTERNAL_ERROR`.

## Out of scope

* KYC document verification
* AML checks
* manual review
* debit card creation
* overdraft setup
* account closure
* customer profile management
* joint accounts
* business accounts
* multi-currency accounts

## Implementation notes

* keep the feature small and local
* keep validation and response mapping explicit
* generate `account_id` inside the feature
* keep internal persistence details outside the external HTTP contract except for the documented `500 INTERNAL_ERROR` result
